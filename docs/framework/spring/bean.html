    Spring Bean的生命周期

    Ryan 6/17/2021 Bean

    在Spring中，那些组成应用程序的主体及由Spring IoC容器所管理的对象，被称之为bean。简单地讲，bean就是由IoC容器初始化、装配及管理的对象，除此之外，bean就与应用程序中的其他对象没有什么区别了。而bean的定义以及bean相互间的依赖关系将通过配置元数据来描述。Spring只帮我们管理单例模式Bean的完整生命周期，对于prototype的bean，spring 在创建好交给使用者之后则不会再管理后续的生命周期。

    先看一下图例

    # 实例化bean对象

    spring中实例化bean对象有三种方法：

    1. # 构造函数

       <bean id="personService" class="cn.mytest.service.impl.PersonServiceBean"></bean>
      1

      这也是最常用的一种实例化方法，id是对象名称，class是要实例化的类，正常调用就可以获取这个bean。

      需要注意的是，这个类里面必须要有一个无参构造器。

    2. # 静态工厂

      public class TestServiceFactory {  
    //创建静态方法  
    public static TestServiceBean createTestBean(){  
         //返回实例化的类的对象  
        return new TestServiceBean();  
    }  
}

    # BeanDefinition

    简而言之，beanDefinition就是把bean实例化出来的一个模型对象，它里面描述了bean是单例还是原型，是否懒加载等等信息。当spring容器启动得时候会去调用 ConfigurationClassPostProcessor 这个后置处理器完成扫描，每扫描到得一个符合规则的类， spring都会实例化一个BeanDefinition对象，然后把根据类的类名生成一个bean的名字 ， 继而spring会把这个beanDefinition对象和生成的beanName放到一个map当中，key=beanName，value=beanDefinition对象 。

    源码当中是这样描述的：

    /**
 * A BeanDefinition describes a bean instance, which has property values,
 * constructor argument values, and further information supplied by
 * concrete implementations.
 *
 * <p>This is just a minimal interface: The main intention is to allow a
 * {@link BeanFactoryPostProcessor} to introspect and modify property values
 * and other bean metadata.
 */

    # Bean工厂后置处理器BeanFactoryPostProcessor

    他的作用就是让我们去操作bean工厂。如果实现了BeanFactoryPostProcessor,会接下来处理bean工厂的后置处理器逻辑。

    实现该接口，可在bean对象创建之前修改它的属性，同时我们也可以在这个阶段进行bean得注入。

    public class MyTest implements BeanFactoryPostProcessor {
    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) throws BeansException {
        beanFactory.registerSingleton("myBean",new T());
    }
}

    当经过上述几个步骤后，bean对象已经被正确构造，但如果你想要对象被使用前再进行一些自定义的处理，就可以通过BeanPostProcessor接口实现。 该接口提供了两个函数：

    • postProcessBeforeInitialzation( Object bean, String beanName ) 当前正在初始化的bean对象会被传递进来，我们就可以对这个bean作任何处理。 这个函数会先于InitialzationBean执行，因此称为前置处理。 所有Aware接口的注入就是在这一步完成的。
    • postProcessAfterInitialzation( Object bean, String beanName ) 当前正在初始化的bean对象会被传递进来，我们就可以对这个bean作任何处理。 这个函数会在InitialzationBean完成后执行，因此称为后置处理。

    # 初始化

    初始化逻辑动作一般是根据业务来决定的，如果实现了InitializingBean 接口，Spring在初始化的过程就回去加载其中的逻辑

    @Component
public class UserService implements InitializingBean {
    public void afterPropertiesSet() throws Exception {

    }
}
    1
    2
    3
    4
    5
    6

    这一阶段也可以在bean正式构造完成前增加我们自定义的逻辑，但它与前置处理不同，由于该函数并不会把当前bean对象传进来，因此在这一步没办法处理对象本身，只能增加一些额外的逻辑。 若要使用它，我们需要让bean实现该接口，并把要增加的逻辑写在该函数中。然后Spring会在前置处理完成后检测当前bean是否实现了该接口，并执行afterPropertiesSet函数。

    当然，Spring为了降低对客户代码的侵入性，给bean的配置提供了init-method属性，该属性指定了在这一阶段需要执行的函数名。Spring便会在初始化阶段执行我们设置的函数。init-method本质上仍然使用了InitializingBean接口。

    # 执行AOP

    开启aop需要添加注解@EnableAspectJAutoProxy。

    AOP后拿到的Bean不是原来的Bean，是代理对象。

    代理对象放到单例池

    单例池中的Bean是放到了一个concurrentHashMap中，key是BeanName，value是代理Bean对象。

    # DisposableBean

    当Bean不再需要时，会经过清理阶段，如果Bean实现了DisposableBean这个接口，会调用其实现的destroy()方法；

    # destroy-method

    如果这个Bean的Spring配置中配置了destroy-method属性，会自动调用其配置的销毁方法。

    bean销毁和spring没什么关系，spring只是在销毁之前做一些业务操作，比如说释放某些资源，或者回写数据库等操作。bean销毁是gc在做 。
